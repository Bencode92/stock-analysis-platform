# TradePulse — Sector & Market Comparator (technical handoff)

## 1. Goal

Add a side-by-side comparator to two existing pages of the platform:

- `secteurs.html` → compare **2 sectors** (each backed by a sector ETF)
- `marches.html`  → compare **2 markets** / country indices (each backed by an index ETF)

For each pick, display the **top 10 holdings of the underlying ETF** (already
ranked by weight) **enriched** with per-stock perf and quality metrics, then
compute weighted aggregates and declare a winner per metric and overall.

User constraints (verbatim, condensed):
- Top holdings must stay sorted by ETF weight (no re-ranking by our filters).
- Aggregates must use the holding weight for weighting (not market cap), to
  reflect what the ETF actually holds.
- Selection UX must be a chip grid (not dropdowns), grouped by index family.
- Visual: heatmap on perf + quality cells; winner cells colored green, loser
  red; global X–Y score banner.
- Must auto-disable items where stock matching coverage is too low
  (`< 4` matched holdings).

## 4. Architecture inside `js/sector-comparator.js`

Single IIFE, four logical layers:

```
SOURCES   ──>  data files + flatten functions per source (sectors / markets)
LOAD      ──>  bootstrapShared() loads etf_holdings + 3 stocks files (cached)
              bootstrap(source) loads source.file and flattens its items
MATCH     ──>  buildStockIndex() + matchStock(holding) — 3-tier ticker/name
COMPUTE   ──>  buildRows(symbol)  → top-10 holdings enriched
              computeAggregates(rows)  → weighted/median metrics
              compareAggregates(L,R)   → win/lose per metric, leftWins/rightWins
RENDER    ──>  injectStyles() + initInstance(containerId, sourceKey)
              renderChips, renderRows, renderAggregates, renderPanel,
              renderComparison
```

### 4.1 Sources config

```js
const SOURCES = {
  sectors: {
    file: 'data/sectors.json',
    noun: 'secteur',
    title: 'Comparateur de secteurs',
    regions: [{key:'all',label:'Tous'}, {key:'europe',label:'Europe'}, {key:'us',label:'US'}],
    flatten: flattenSectors,
  },
  markets: {
    file: 'data/markets.json',
    noun: 'marché',
    title: 'Comparateur de marchés',
    regions: [
      {key:'all', label:'Tous'},
      {key:'europe', label:'Europe'},
      {key:'north-america', label:'Amérique du Nord'},
      {key:'asia', label:'Asie'},
      {key:'latin-america', label:'Amérique latine'},
      {key:'other', label:'Autres'},
    ],
    flatten: flattenMarkets,
  },
};
```

`flattenSectors` / `flattenMarkets` produce a uniform `Item`:
```ts
{
  key: string,         // unique, "<category>::<symbol>"
  label: string,       // long label, used as tooltip
  shortLabel: string,  // chip text, post-disambiguation
  family: string,      // index family / region label, used as chip group header
  region: string,      // human-readable
  regionKey: string,   // raw key from JSON, used by region tab filter
  symbol: string,      // ETF ticker, used to look up etf_holdings.etfs[symbol]
  category: string,
  sectorObj: object,   // raw entry, kept for future needs
  // populated post-bootstrap:
  coverage: number,        // matched holdings out of totalHoldings
  totalHoldings: number,
  disabled: boolean,       // coverage < MIN_COVERAGE (=4)
}
```

After flattening we run two post-processing passes:

1. `disambiguateLabels(items)` — when several items in the same `family` share
   the same `shortLabel` (typical case: 5 different US ETFs all with country =
   "Etats-Unis"), append a discriminator extracted from the long label via a
   regex matching common index names (`S&P 500`, `NASDAQ`, `Russell 2000`,
   `MSCI Xxx`, `FTSE`, `DAX`, `CAC 40`, `Nikkei`, `Hang Seng`, `Dow Jones`),
   falling back to the ETF symbol if no match.
2. `computeCoverage(items)` — for each item, take its first 10 ETF holdings
   and count how many resolve through `matchStock`. Stores `coverage`,
   `totalHoldings`, and a boolean `disabled = coverage < MIN_COVERAGE`.

### 4.2 Stock matching (the part most likely to matter for an expert)

Single in-memory `Map` built once from all 3 stock files:

```js
function buildStockIndex(stockSets) {
  const idx = new Map();
  const add = (k, v) => { if (k && !idx.has(k)) idx.set(k, v); };
  stockSets.forEach(set => (set?.stocks || []).forEach(stk => {
    add(`T:${(stk.ticker || '').toUpperCase()}`, stk);
    add(`T:${stripSuffix(stk.ticker)}`, stk);
    if (stk.resolved_symbol) {
      add(`T:${stk.resolved_symbol.toUpperCase()}`, stk);
      add(`T:${stripSuffix(stk.resolved_symbol)}`, stk);
    }
    const n1 = normalizeName(stk.name);
    const n2 = normalizeName(stk.name_api);
    if (n1) add(`N:${n1}`, stk);
    if (n2) add(`N:${n2}`, stk);
  }));
  return idx;
}
```

`stripSuffix("AAPL.O") -> "AAPL"`. `normalizeName` lowercases, removes
punctuation, and strips legal-form noise: `the corp corporation inc
incorporated company co ltd limited plc sa ag nv se ab oyj holdings group tr
trust class a/b/c`.

Lookup priority for a holding:
1. `T:<symbol upper>`
2. `T:<symbol stripped>`
3. `N:<normalized holding.name>`

### 4.3 Aggregates and winner logic

`AGG_METRICS` is the canonical ordered list driving both rendering and the
winner-by-metric comparison:

```js
const AGG_METRICS = [
  { key: 'perf_ytd',   label: 'Perf YTD (pondérée)',  dir:  1, fmt: ... },
  { key: 'perf_1y',    label: 'Perf 1Y (pondérée)',   dir:  1, fmt: ... },
  { key: 'perf_3y',    label: 'Perf 3Y (pondérée)',   dir:  1, fmt: ... },
  { key: 'quality',    label: 'Quality (pondéré)',    dir:  1, fmt: ... },
  { key: 'buffett',    label: 'Buffett (pondéré)',    dir:  1, fmt: ... },
  { key: 'roe',        label: 'ROE médian',           dir:  1, fmt: ... },
  { key: 'roic',       label: 'ROIC médian',          dir:  1, fmt: ... },
  { key: 'net_margin', label: 'Net margin médiane',   dir:  1, fmt: ... },
  { key: 'fcf_yield',  label: 'FCF yield médian',     dir:  1, fmt: ... },
  { key: 'vol_3y',     label: 'Vol 3Y (pondérée)',    dir: -1, fmt: ... },
  { key: 'max_dd_3y',  label: 'Max DD 3Y (pondéré)',  dir:  1, fmt: ... },
  { key: 'coverage',   label: 'Couverture stocks',    dir:  0, fmt: ... },
];
```

`dir: 1` = higher is better, `dir: -1` = lower is better, `dir: 0` = neutral
(not scored). `compareAggregates(aggL, aggR)` walks this list, increments
`leftWins`/`rightWins`, and tags each cell `'win' | 'lose' | 'tie' | 'neutral'`.

`computeAggregates(rows)` (inputs are top-10 ETF holdings, only matched
stocks count):

```js
{
  coverage,
  total,
  perf_ytd:  weightedAvg(matched, r => r.stock.perf_ytd),  // weight = holding.weight
  perf_1y:   weightedAvg(... perf_1y),
  perf_3y:   weightedAvg(... perf_3y),
  quality:   weightedAvg(... quality_score),
  buffett:   weightedAvg(... buffett_score),
  roe:       median(... roe),
  roic:      median(... roic),
  net_margin:median(... net_margin),
  fcf_yield: median(... fcf_yield),
  vol_3y:    weightedAvg(... volatility_3y),
  max_dd_3y: weightedAvg(... max_drawdown_3y),
  beta:      weightedAvg(... beta),  // computed but not displayed yet
}
```

### 4.4 Render

- **Chips** are grouped by `family` in a card (`.sc-family`), each chip shows
  `shortLabel` and a `coverage / totalHoldings` badge. Disabled chips get
  `is-disabled` class (line-through, opacity .35, ignored on click).
- **Region tabs** filter by `regionKey`.
- **Selection** is a 2-slot FIFO (`selected` array). Picking a third chip
  drops the oldest. Comparison renders automatically as soon as both slots
  are filled.
- **Panel** renders the title, an "X victoires sur N critères" headline, the
  top-10 table, then the aggregates grid. Heatmap backgrounds are inline
  styles computed by `heat(value, min, max)`.
- **Banner** above the two panels announces the overall winner with the
  numeric score (`X–Y`).

### 4.5 Styles

All CSS is injected at runtime by `injectStyles()` (single `<style id="sc-styles">`)
to avoid touching the host page's stylesheets. Class prefix is `sc-`. The
section uses `width: 100%` (no max-width) and inherits the parent container
width, currently giving the two panels ~50% each on desktop, stacking below
1000px.

## 5. Auto-init flow

```
DOMContentLoaded
  └── init()
        ├── if #sector-comparator exists: initInstance('sector-comparator', 'sectors')
        └── if #market-comparator exists: initInstance('market-comparator', 'markets')

initInstance(id, sourceKey):
  1. inject styles (idempotent)
  2. render skeleton (title, hint, region tabs, chips container, actions, result container)
  3. await bootstrap(source)            // shared cache + flatten + disambiguate + coverage
  4. wire chip click, region tab click, clear button
  5. refresh() → renderChips + render comparison if 2 selected
```

`bootstrapShared` is memoized via `shared.loading` so the heavy files load
once even if both containers were on the same page.

## 6. Known issues / open questions for the expert

These are the points I'd love a second opinion on:

### 6.1 Stock matching coverage on Asia / non-US

- US sector ETFs: coverage is typically 9–10/10.
- Europe: 8–10/10 (the legal-form stripping handles most cases).
- **Asia is the weak spot.** Holdings carry local symbols like `2330`
  (TSMC, Taiwan), `005930` (Samsung, KR), `0700` (Tencent, HK). Our
  `stocks_asia.json` may use ADR tickers or suffixed local symbols. As a
  result several Asian markets are flagged disabled.
- Question: should we (a) preprocess `stocks_asia.json` to add an alias
  index by local code, (b) maintain a manual override map for the worst
  offenders, or (c) build a more aggressive name-similarity match (e.g.
  token Jaccard) at index-build time?

### 6.2 Aggregate methodology

- Perf and quality are **weighted by holding weight**. ROE/ROIC/net margin/
  FCF yield use **median** to avoid being dominated by one mega-cap. Vol and
  max DD are weighted. Is this the right mix, or should everything be
  consistent (all weighted, or all median)?
- Top 10 only. The ETF API config caps at 10 holdings; the long tail is
  invisible. For broad-index ETFs (e.g. NASDAQ 100) the top 10 may only
  represent ~50% of weight, so aggregates can be misleading. Should we
  display the cumulative weight covered (`top_weight` is in the data) and
  warn when it is too low?
- "Winner per metric" is binary. A 0.1% difference counts the same as a
  20% difference. Is this acceptable, or should we add a tie zone
  (epsilon based on metric scale)?

### 6.3 Heatmap ranges

The min/max ranges are hardcoded: `perf_ytd ±20%`, `perf_1y -20%/+40%`,
`quality 30/85`, etc. They were tuned by eye on the current dataset. They
should probably be data-driven (e.g. p10/p90 of the visible top-10s) or at
least documented and tunable from config.

### 6.4 Disambiguation regex fragility

`disambiguateLabels` extracts an index name from the long label using a
regex over a hand-maintained list of patterns. New index families won't
match and fall back to the ETF symbol. A cleaner approach would be to
read `indexFamily` (sectors.json has it) or a dedicated `index_short`
field if `markets.json` could expose one.

### 6.5 Region tabs vs `regionKey`

- Sectors store `regionKey = (s.region||'').toLowerCase()`, so `'europe'` or `'us'`.
- Markets store the raw JSON region key (`'europe'`, `'north-america'`, …).
- The two tab sets diverge on purpose. If `sectors.json` ever gains an Asia
  entry the sector tabs won't expose it. Should we always derive tabs from
  the actual data instead of hardcoding them in `SOURCES`?

### 6.6 Performance

- Total payload at first compare: ~3.5 MB JSON (mostly `stocks_us.json`).
  Loaded with `Promise.all`, parsed in main thread. On a slow connection
  the user sees "Chargement…" for 1–3 s. Worth a streaming or chunked
  approach? A pre-computed `stocks_min.json` with only the fields we use
  would cut this to <300 KB.
- The stock index Map currently has ~12k entries. Build time is fine
  (<50 ms). No issue today, just noting for scale.

### 6.7 No tests

There are no unit tests for `matchStock`, `computeAggregates`,
`compareAggregates`, or `disambiguateLabels`. These are the functions
where a regression would silently produce wrong numbers. Recommended
priority for the expert: at minimum, golden tests on a frozen subset of
`etf_holdings.json` + `stocks_*.json`.
